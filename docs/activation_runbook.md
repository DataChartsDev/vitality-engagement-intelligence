# Offline Activation Runbook

## Purpose

This runbook covers the local Stage 5 activation workflow.

The command verifies scoring and contact-context artifacts, applies deterministic policy rules, and writes local review artifacts.

It does not upload to BigQuery, contact members, dispatch interventions, or approve records for outreach.

## Preconditions

Before running the command, confirm:

1. The virtual environment is active.
2. The repository has no unexpected changes.
3. The scoring artifact was generated by this repository.
4. The contact-context files come from an approved legitimate source.
5. The context snapshot is not later than the decision timestamp.
6. The decision timestamp includes a timezone offset.
7. Input and output paths do not overlap.
8. Selected records will receive authorised human review.

Do not fabricate a production contact-context artifact merely to make the command run.

## Required files

The CLI requires a contact-context Parquet file and its matching metadata JSON. There are intentionally no default production contact-context paths.

The default verified scoring files are:

- `artifacts/scoring/python_logistic_scoring_predictions.parquet`
- `artifacts/scoring/python_logistic_scoring_predictions.metadata.json`

## Local command

Run only after approved contact-context files exist:

    python -m vitality_engagement.activation.cli `
        --context-path "C:\approved-inputs\member_contact_context.parquet" `
        --context-metadata-path "C:\approved-inputs\member_contact_context.metadata.json" `
        --decision-timestamp "2026-07-21T08:00:00+02:00"

The paths and timestamp above are examples. Do not execute the example unchanged.

## Expected outputs

- `artifacts/activation/activation_decisions.parquet`
- `artifacts/activation/activation_decisions.metadata.json`
- `artifacts/activation/human_review_queue.parquet`
- `artifacts/activation/human_review_queue.metadata.json`

The review queue contains only records already marked `selected_for_review`, ordered by deterministic priority rank. Every row has the fixed status `pending_human_review`; the workflow does not create an approved-for-contact status.

The review-queue metadata records SHA-256 digests for both source activation artifacts, so any source change invalidates verification.

A successful command prints the run ID, activation artifact paths, review-queue paths, source rows audited, records selected for human review, and confirmation that execution was local-only.

## Fail-closed behaviour

The workflow rejects missing files, digest mismatches, invalid metadata, unexpected columns, duplicate context member IDs, naive or future timestamps, invalid threshold classifications, input/output path collisions, and unsafe policy settings.

Verification failures occur before final activation outputs are written.

## Review checklist

Before downstream persistence, confirm that every scoring row has one audit outcome, scoring and contact-context lineage are present, timestamps are correct, selected ranks are contiguous, the selected count does not exceed capacity, metadata counts match Parquet outcomes, and human and governance approval has been recorded.

Also verify that the review queue contains only `selected_for_review` records, every queue row remains `pending_human_review`, and the review-queue source digests match the verified activation Parquet and metadata files.

## Warehouse persistence

The warehouse uploader is separate and is not available through the local CLI.

Do not upload merely to demonstrate that the uploader works.

A successful local run does not authorise outreach, messaging, eligibility changes, penalties, treatment assignment, or clinical conclusions.
